## 一、我们是谁

**平台定位**
手机厂商的游戏分发业务整体。不局限于应用商店、游戏中心——这两者是最具代表性的流量场景，但不是全部。核心业务：为手机用户提供游戏分发下载，以及游戏中的一系列增值服务，让用户在渠道平台玩得更爽，平台赚更多钱，实现双赢。

**硬件入口优势**
作为手机厂商，我们拥有系统级入口和设备数据，这是纯第三方平台不具备的。但这个优势是否被充分利用，是持续需要审视的问题。

**当前阶段**
AI能力从0到1建设期，已初步探索建设：
- 底层用户理解：user-game-weight-tag-score-profile（语义化）-figure
- 个性化query引导语生成
- game_insight（游戏结构理解）
- user_insight（用户偏好三层推理）
- Compare Engine（多游对比+rejectList负决策解释）
- 搜索意图识别（Find/Understand/Decide链路）
- 小游戏schema体系（在建）

**我们不是什么**
- 不是游戏厂商，不控制游戏内容和更新节奏
- 不是内容社区，无UGC积累优势
- 商业逻辑是分发+增值服务，不是内容订阅

## 三、竞品分类与分析视角

威胁逻辑的核心是**流量抢夺路径**，不是功能相似度。分三类：

---

**第一类 · 流量场景直接竞争（高威胁）**
在游戏下载分发这个核心场景上与我们争用户。用户的游戏消费闭环（发现→决策→下载→付费）被留在对方平台完成，不经过我们。

代表产品：抖音、快手（内嵌小游戏/游戏分发）、微信小游戏、官渠（游戏厂商自建渠道）、TapTap

分析重点：
- 他们的分发规模和增速
- 对游戏厂商的让利政策变化（影响我们的供给侧）
- 是否在侵蚀我们的核心用户群（哪类用户在流失）

---

**第二类 · 决策加工替代（中威胁）**
用户在我们平台外完成了"玩哪款/要不要玩"的决策，来我们这里只是执行下载。我们从决策入口退化为工具入口，长期损伤平台价值。

代表产品：小红书（问一问/游戏攻略）、豆包、B站（游戏区/AI功能）、抖音搜索

分析重点：
- 他们在游戏决策场景的能力演进（搜索、推荐、AI问答）
- 用户在这些平台完成决策后的跳转路径
- 我们能否在他们触达用户之前或之后插入决策能力

---

**第三类 · 学习参考对象（非威胁，重点看亮点）**
与我们定位不直接竞争，但在AI能力、产品体验、商业模式上有值得借鉴的创新。

代表产品：Google Play、Steam、Roblox、App Store、Perplexity（游戏场景）、逗逗伴游等AI伴游产品、华为/小米游戏中心（同类平台）、好游快爆

分析重点（换视角）：
- 他们解决了什么用户问题，用了什么方法
- 哪些能力/设计可以被我们移植或改造
- 对我们当前在建模块有哪些启发（game_insight/user_insight/搜索意图等）

## 四、分析框架：看到竞品动态，沿什么维度判断

**先判断类别，再选对应框架。**

---

**第一类动态分析框架（流量竞争）**
```
1. 是什么？（功能/政策/数据，信息源+日期）
2. 抢的是哪个场景的流量？（发现/下载/付费/留存）
3. 规模信号：有没有数据佐证（DAU/下载量/厂商合作动态）
4. 我们的对应场景现状如何？
5. 判断：需要防守/跟进/差异化，优先级高/中/低
```

**第二类动态分析框架（决策替代）**
```
1. 是什么？（功能/体验，信息源+日期）
2. 在游戏决策链路的哪个节点切入？（找游戏/比较/攻略/付费决策）
3. 用户路径：用完之后去哪？是否跳出到我们平台？
4. 与我们在建能力的重叠度（game_insight/user_insight/搜索意图）
5. 判断：对我们决策入口价值的侵蚀程度，建议动作
```

**第三类动态分析框架（学习参考）**
```
1. 是什么？（功能/设计，信息源+日期）
2. 它解决了什么用户问题？用了什么方法？
3. 哪个设计/能力值得移植？对应我们哪个在建模块？
4. 移植的前提条件和主要障碍是什么？
5. 建议：直接参考/改造后用/仅作信号，优先级
```

## 六、输出标准

每份报告三层结构，缺一不可：

**Layer 1 · 信息层**
- 发生了什么
- 信息来源（官方博客/媒体报道/用户反馈）+ 日期
- 可信度标注（官方确认 / 媒体推断 / 社区传言）

**Layer 2 · 分析层**
- 它解决了什么问题
- 用四维框架（见第四节）判断位置和差距

**Layer 3 · 判断层**
- 对我们意味着什么（威胁/机会/信号）
- 建议动作（加速/差异化/观察/忽略）
- 优先级：高/中/低

**格式要求**
- 结论前置，判断层不得用模糊语言（"可能""也许""值得关注"不算判断）
- 数据必须标注来源，推断必须标注"推断"
- 每条竞品动态300-500字
- 周报总结单独一页，结论不超过5条，不限字数但每条必须有明确动作建议
- 每条借鉴建议末尾必须注明「执行前提」：列出该建议依赖的我方能力/供给/数据条件，若不具备需标注

## 七、交付前自检（必须过，不得跳过）

### A · 结构完整性
- [ ] 每一条核心判断，能在正文中找到对应的支撑段落？若无，降级为「待验证假设」并明确标注
- [ ] 所有「推断」已明确标注 `[推断]`？
- [ ] 信息盲区板块是否包含三类：①外部数据缺失 ②我方内部数据缺失 ③**行为层面缺失**（用户实际行为重叠/迁移，不只是指标层面）
- [ ] 所有建议动作已按「紧迫性 × 我方执行能力」给出优先级排序？

### B · 推理链完整性
对每一个「建议动作」或「借鉴机制」，确认以下三项均已说明：
- [ ] **触发条件**：在什么情况下这个建议成立？
- [ ] **执行前提**：这个建议依赖哪些我方已有/未有的能力、供给或数据条件？
- [ ] **接驳点**：借鉴建议是否已映射到我方具体系统/框架/节点（不能只说"有启发"或"直接相关"）

### C · 威胁判断质量
- [ ] 威胁路径的逻辑是否考虑了"同时分发"的可能性？（厂商选择竞品 ≠ 放弃我们）
- [ ] 「时间/注意力争夺」类论断是否基于**行为习惯变化**而非零和时长假设？
- [ ] 威胁分级是否区分了「已发生」vs「趋势信号」vs「潜在风险/待验证假设」？

## 八、信息采集规范

每次生成报告前，按以下搜索词和来源优先级采集信息。**采集方式取决于当前环境是否具备联网搜索能力**：具备时用任意可用的网页搜索工具主动采集；不具备时请用户粘贴素材或提供来源 URL（见 SKILL.md 第二步的降级说明）。搜索结果必须标注来源URL和发布日期，无法确认日期的内容不得使用。

### 8.1 第一类竞品搜索词模板
```
抖音小游戏 新功能/政策/数据 [当前年月]
快手小游戏 游戏分发 [当前年月]
微信小游戏 新游/政策/开发者 [当前年月]
TapTap 新功能/AI/推荐 [当前年月]
游戏厂商 官渠 自建渠道 [当前年月]
```

### 8.2 第二类竞品搜索词模板
```
小红书 游戏 AI搜索/推荐/攻略 [当前年月]
豆包 游戏 功能 [当前年月]
B站 游戏区 AI [当前年月]
```

### 8.3 第三类竞品搜索词模板
```
Google Play 新功能/AI推荐 [当前年月]
Steam 新功能/Discovery [当前年月]
好游快爆 新功能 [当前年月]
AI游戏伴游 新产品 [当前年月]
```

### 8.4 AI行业信号搜索词模板
```
AI游戏 大模型 新进展 [当前年月]
游戏AI 米哈游/网易/腾讯 [当前年月]
游戏推荐 AI算法 [当前年月]
```

### 8.5 来源优先级
1. **官方一手来源**：平台官方博客、开发者文档、发布会官方报道 → 标注`[官方]`
2. **权威行业媒体**：36氪、GameLook、葡萄君、17173、新浪财经 → 标注`[媒体·来源名]`
3. **第三方数据机构**：QuestMobile、极光月狐、DataEye → 标注`[机构·来源名]`
4. **分析解读文章**：知乎、虎嗅等 → 标注`[解读·来源名]`，可信度需额外说明
5. **禁止使用**：无法确认发布日期的内容、匿名论坛帖子、无来源数据

### 8.6 信息时效要求
- 周报：只收录距发布日**14天内**的动态，超过14天的降级为背景信息
- 专题报告：核心数据允许使用3个月内的报告，需标注数据采集时间

## 九、两种报告形态与触发规则

### 9.1 周报

**触发**：用户主动请求（"出竞品周报"等触发词）。若运行环境支持定时任务，可配置为每周一上午自动运行。

**结构**：
```
封面：周次 + 日期 + 本周最高优先级信号（1句话）
第一类动态：3-5条，每条300-500字，三层结构
第二类动态：2-3条，每条300-500字，三层结构
第三类动态：1-2条，有亮点才写
AI行业信号：1-2条简报
本周总结：结论不超过5条，每条必须有明确动作建议
```

**交付格式**：HTML（对齐baseline视觉标准）

---

### 9.2 专题报告（每两周一篇，有重大动态可提前触发）

**触发条件**（满足任一即触发）：
- 第一类竞品有重大功能/政策发布（影响分发规模或供给侧）
- 第二类竞品AI能力有显著跃升（影响决策入口争夺）
- 老板/业务侧提出具体竞品问题
- 固定：每两周轮转一个竞品做深度

**结构（固定六节）**：
```
核心判断（结论前置）
01 · 产品概览（定位+规模数据+现阶段定性）
02 · 用户与场景（画像+典型路径+关键数据）
03 · 产品能力拆解（核心功能+差异化机制+已知短板）
04 · 商业结构（变现路径+对厂商价值主张）
05 · 综合评估（星级评估表+优势+风险）
06A · 威胁视角（威胁路径+升级信号+我们的弱势+建议动作）
06B · 学习视角（可移植机制，每条必须含执行前提）
行动排序（P0/P1/P2优先级矩阵）
信息盲区（三类：外部/内部/行为层面）
来源说明（全部可点击溯源）
```

**交付格式**：HTML（侧边栏导航+可点击来源标注，对齐baseline）

- 小红书游戏生态

## 十、执行指引

> 以下为通用执行流程，不依赖任何特定平台或定时调度环境。若运行环境恰好支持定时任务，可把"周报"流程挂到每周一自动运行；否则按用户请求手动执行即可。

### 周报执行流程
```
1. 读取本文档作为分析视角
2. 信息采集：
   · 环境有联网搜索 → 按第八节搜索词模板采集本周动态
   · 环境无联网搜索 → 请用户粘贴素材/提供来源URL
3. 过滤：只保留14天内的内容，标注来源和日期
4. 按三类竞品框架分析，每条动态走对应分析框架
5. 执行第七节自检（A/B/C三类）
6. 生成HTML报告，文件名格式：竞品周报_W[周次]_[年份].html
7. 交付给用户（如需存档，由用户指定保存位置）
```

### 专题执行流程
```
触发：用户请求，或固定每两周一篇
1. 读取本文档作为分析视角
2. 确定本期专题对象（见9.2轮转计划）
3. 信息采集（同上：有搜索能力则深度搜索，官方来源优先；否则用户提供素材）
4. 按六节结构撰写，信息层全部附链接
5. 执行第七节自检
6. 生成HTML报告，文件名格式：专题报告_[竞品名]_[年份W周次].html
7. 交付给用户
```

### 质量红线（任何情况下不得违反）
- 数据无来源URL → 不得写入报告
- 来源无法确认日期 → 不得使用
- 核心判断无正文支撑 → 必须降级为「待验证假设」
- 判断层出现"可能""也许""值得关注"等模糊语言 → 必须重写
- 借鉴建议无「执行前提」 → 不得交付
